LBI: The Loolia Brand Index

  1. Home

Transforming the LBI from a week-long task into a two-click job.

  • Software Engineer
  • Individual
  • 12 months
  • Loolia

Automating the process of calculating the Loolia Brand Index by building two independent applications, one of which is the main Loolia calculator while the other is a product entry application used to log product appearances on videos.

The Problem

Loolia is an online platform and shop that aim on respectively promoting and selling various beauty products, notcably throught their Youtube channel that promotes multiple brands. The Loolia Brand Index (LBI) is a ranking system that quantifies the popularity of these brands. However, the process of calculating the LBI is currently manual, tedious, and time-consuming. Additionally, the current LBI calculation method does not account for retention rates, making it less precise.

The Solution

The solution is to build two independent applications, one of which is the main Loolia calculator that will compute and display the LBI of all categories at multiple dates, and the other is a product entry application used by the production team to log product appearances on videos in order to source the necessary data to the main application. By automating the LBI calculation process, the project aims to save time, increase accuracy, and eliminate errors. The system also has provisions for future extensions and tuning of the index.

Ultimately, this implementation helped take a manual process that would take over a week to a collective effort that requires a few clicks from each person. The wok is not only faster and more collaborative, it is also more accurate as the data is richer and in-real time.

My Responsabilities

As a software engineer, my responsibilities include developing both applications using Java and Java Swing UI. I am also responsible for designing the database and linking it to the applications using JDBC. I will ensure that both applications are properly integrated to communicate and exchange data seamlessly. Additionally, I will handle the project’s dependencies using Maven and Gradle for compilation, and ensure that the project’s folder and package architecture is well-structured.

Challenges

Challenge 1: Incorporating Retention Rates Another challenge is incorporating retention rates into the LBI calculation. Retention rates, in this context, measure the percentage of users who continue to watch a product video over a specific period. Retention rates are important because they reflect which products were the most looked at, which is an essential factor in determining the long-term success of a product campaign. However, calculating retention rates can be difficult. On the one hand, getting this level of granularity in data require a special type of Youtube API to be set up to cater to security concerns as well as volumes of queries. On the other hand, incorporating retention rates into the LBI calculation significantly increases the complexity of the operations behind it. Getting to know the Youtube API was a journey but likely the presentation and tools out there is quite rich. The biggest hurdle at this step was that Google changed their policies around API and privacy mid-way through the project which involved being forced to migrate to their API v3. Moreover, I looked at the increase in computational complexity which impacts performance. Besides the normal code optimisation in Java and query optimisations in SQL, we hit the limit imposed by the lag of the API and the low specs of the machine the code is run on. Taking into consideration the low-frequency usage of this bit of the code as well as its ability to run off-peak times, it was not deemed necessary to invest more time in resolving a non-issue.

Challenge 2: Encourage Adoption As the initial data entry and calculation was done by a single employee, it was not easy to convince other employee to participate. The new design requires video editors to enter the product included in the videos they produce as well as entering the timestamps at which they appear. Ultimately, the solution is to eliminate any friction between the user and the data entry. To this end, I tackled the problem with 2 things in mind: Platform-Independence The project requires that the applications be platform-independent, which means that they must work seamlessly on all operating systems. Achieving platform-independence can be a challenge because different operating systems have different architectures, libraries, and dependencies. Ensuring that the application works smoothly on all operating systems may require significant development resources and extensive testing. User-Friendly Interface Finally, ensuring that the applications are user-friendly with a simple and intuitive user interface helped making the task easier and faster. The success of thius application depends on the ease of use and accessibility to the end-users. Developing a user-friendly interface requires a deep understanding of human-computer interaction, usability engineering, and design thinking. Therefore, designing an interface that is intuitive and easy to use can be a challenging task that requires a lot of creativity and expertise.
